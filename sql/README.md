# SQL: Database Schema & Recurring Extract Queries

## Overview

SQL schema and reusable queries for fund performance monitoring, portfolio analysis, and data quality governance. Designed to run against the SQLite database (`fund_data.db`) produced by the Python pipeline.

These queries represent the "recurring data extracts" that a PDS team would run on a regular basis for LP reporting, internal monitoring, and GP data quality tracking.

## File Structure

```
sql/
  schema.sql                       # Database table definitions
  queries/
    fund_performance.sql           # Fund-level performance metrics
    portfolio_summary.sql          # Portfolio breakdowns by vintage, strategy, status
    data_quality.sql               # Data quality monitoring and GP comparison
```

## Database Schema

The database contains 3 tables:

### investments
The core table holding standardized fund data from all GPs.

| Column | Type | Description |
|--------|------|-------------|
| investment_id | TEXT (PK) | Unique identifier per investment |
| fund_name | TEXT | Fund vehicle name |
| company_name | TEXT | Portfolio company name |
| commitment_eur | REAL | Total commitment in EUR |
| called_eur | REAL | Capital called to date in EUR |
| distributed_eur | REAL | Distributions to date in EUR |
| nav_date | TEXT | NAV reporting date |
| currency | TEXT | Reporting currency (always EUR after conversion) |
| original_currency | TEXT | Original currency before conversion |
| vintage_year | INTEGER | Year of investment |
| gp_name | TEXT | General partner name |
| strategy | TEXT | Investment strategy (Growth Equity, Venture Capital, Buyout) |
| status | TEXT | Investment status (Active, Realized, Written Off) |
| source_gp | TEXT | Source GP identifier |
| source_file | TEXT | Original filename |

### validation_issues
Tracks data quality issues found during validation.

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER (PK) | Auto-incrementing ID |
| row_num | INTEGER | Row number in source file |
| investment_id | TEXT | Affected investment |
| check_type | TEXT | Type of check (Missing Field, Invalid Value, Format Error, Duplicate, Business Rule) |
| field | TEXT | Field with the issue |
| issue | TEXT | Description of the problem |
| severity | TEXT | Critical or Warning |
| source_file | TEXT | Original filename |

### ingestion_log
Audit trail for file processing.

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER (PK) | Auto-incrementing ID |
| filename | TEXT | Source filename |
| gp_source | TEXT | Detected GP source |
| rows_ingested | INTEGER | Number of rows read |
| columns | INTEGER | Number of columns |
| processed_at | TIMESTAMP | Processing timestamp |

## Queries

### fund_performance.sql
Recurring extract for quarterly LP reporting. Produces fund-level metrics grouped by fund and GP: investment count, active/realized split, total commitment, called, distributed, call rate, DPI, vintage range.

### portfolio_summary.sql
Three breakdowns for portfolio allocation reporting:
- **By vintage year**: commitment, called, distributed, call rate, DPI per vintage
- **By strategy**: investment count, GP count, commitment, average call rate per strategy
- **By status**: count, commitment, distributed, average DPI per status

### data_quality.sql
Five queries for internal governance:
- Issue summary by severity
- Issues by check type and severity
- Issues by source file (GP data quality comparison)
- Investments with multiple issues
- Ingestion audit trail

## How to Run

These queries can be run against the `fund_data.db` file generated by the Python pipeline:

```bash
# Using SQLite command line
sqlite3 python/output/fund_data.db < sql/queries/fund_performance.sql

# Or open in any SQLite GUI tool (DB Browser for SQLite, DBeaver, etc.)
```
